# ADR-003 — Contextual Retrieval over classic RAG

**Status:** Accepted
**Date:** May 2026
**Author:** David Font Munoz

---

## Context

The system needs to retrieve relevant operational runbooks and historical postmortems so that Agent 3 (Diagnostic Reasoner) can diagnose the root cause of an incident. Retrieval quality directly impacts diagnostic precision -- an irrelevant retrieved chunk can lead the model to an incorrect hypothesis.

The initial implementation used classic RAG: split documents into 512-token chunks, generate embeddings with sentence-transformers, and retrieve by cosine similarity. During testing a systematic problem was identified.

---

## The problem with classic RAG in operational runbooks

Technical runbooks have a dense hierarchical structure. A 512-token chunk extracted from the "Remediation" section of a DB connection pool runbook might contain:

```
1. Check active queries in PostgreSQL:
   SELECT pid, now() - pg_stat_activity.query_start AS duration
   FROM pg_stat_activity WHERE state != 'idle'
2. Kill queries blocked for > 60s
3. If it persists: kubectl rollout restart deployment/payment-service
```

This chunk, without context, does not indicate which service it affects, what symptoms precede it, or what root cause it addresses. When fragmented it loses the document context that gives it meaning. The resulting embedding is generic -- "restart kubernetes deployment" -- and may be retrieved for unrelated incidents.

The problem is amplified in historical postmortems, where each section (timeline, root cause, lessons learned) has more value when the model knows it belongs to a specific incident for a specific service.

---

## Decision

**Contextual Retrieval** -- technique published by Anthropic in September 2024.

Before generating the embedding for each chunk, 50-100 tokens of context are generated by an LLM that describes what part of the document the chunk is and why it is relevant. The embedding is generated over `context + chunk`, not just the chunk.

```
[GENERATED CONTEXT] This chunk describes the immediate remediation steps
for DB connection pool exhaustion in Python services with PostgreSQL,
specifically how to identify and kill blocked queries and when to
restart the service.

[ORIGINAL CHUNK] 1. Check active queries in PostgreSQL:
SELECT pid, now() - pg_stat_activity.query_start...
```

---

## Implementation

```python
def _add_context_to_chunk(document: str, chunk: str) -> str:
    prompt = (
        f"<document>{document[:2000]}</document>\n"
        f"<chunk>{chunk}</chunk>\n"
        "In 1-2 sentences, explain what this chunk is about "
        "within the document context. Be specific."
    )
    response = llm.invoke([HumanMessage(content=prompt)])
    return f"{response.content}\n\n{chunk}"
```

Context is generated at ingestion time (one-time per document), not at retrieval time. The additional cost is one-off, not recurring.

---

## Reasons

**Measured precision improvement.** According to benchmarks published by Anthropic, Contextual Retrieval reduces retrieval errors by up to 67% compared to classic RAG in dense technical documents. For operational runbooks, where context specificity is critical, the impact is especially significant.

**Acceptable cost.** The additional cost occurs only at ingestion -- when a new runbook or postmortem is added to the knowledge base. At retrieval time (every time Agent 3 queries the RAG) there is no additional cost. A knowledge base of 50 runbooks averaging 5 chunks each requires 250 additional LLM calls at ingestion, a one-time cost amortized indefinitely.

**Impact on diagnosis.** An incorrectly retrieved chunk can lead Agent 3 to a wrong hypothesis that propagates through the entire pipeline -- incorrect remediation, unnecessary HITL, postmortem with wrong root cause. The cost of a false diagnosis in production far outweighs the cost of contextualization.

---

## Discarded alternatives

**Classic RAG (chunk -> embedding -> retrieval).** Initial implementation. Discarded due to the context problems described above.

**HyDE (Hypothetical Document Embeddings).** Generates a hypothetical document for the query before searching. Adds latency at retrieval time (not ingestion) and requires an LLM for every search. Not appropriate for a system where retrieval occurs in real time during an incident.

**Reranking only.** Using Cohere Rerank on classic RAG results improves ordering but does not fix the underlying problem -- chunks without context still generate weak embeddings. Cohere Rerank is implemented as an optional additional layer on top of Contextual Retrieval, not as a substitute.

**Parent document retrieval.** Retrieving the full document instead of chunks. Not scalable with long runbooks -- the LLM context fills up quickly with irrelevant complete documents.

---

## Trade-offs accepted

**Ingestion cost.** Each new document requires N LLM calls to contextualize its chunks (N = number of chunks). With Claude Haiku in production the cost is minimal but exists. Offset by improved diagnostic precision.

**Ingestion latency.** Ingesting a 3,000-word runbook takes ~30 seconds instead of ~2 seconds with classic RAG. Acceptable because ingestion is a maintenance operation, not real-time.

**Dependency on generated context quality.** If the LLM generates incorrect or overly generic context, the resulting embedding may be worse than the chunk alone. In practice this is rare with Haiku/Groq for this specific task, but it is a failure point to monitor.

---

## Consequences

- The 5 example runbooks in `rag/seed_runbooks.py` are ingested with Contextual Retrieval
- Postmortems generated by Agent 5 are automatically ingested with Contextual Retrieval (learning loop)
- Retrieval uses dense search + optional Cohere Rerank as a second improvement layer
